================================================================================
PERFORMANCE ANALYSIS
================================================================================

1. CHUNKING PHASE (30.85 seconds for 50 PDFs)
   - Operation: PDF text extraction + recursive character splitting
   - Input: 911 pages, ~500k tokens
   - Output: 4,596 chunks, 800 tokens/chunk, 100 overlap
   - Speed: 149 chunks/second
   - Bottleneck: I/O bound (PDF loading)

2. EMBEDDING GENERATION (205.34 seconds for 50 PDFs)
   - Operation: Sentence-transformers encoding (all-MiniLM-L6-v2)
   - Model: 384-dimensional embeddings
   - Batch size: 32 chunks per batch
   - Total embeddings: 4,596 embeddings
   - Speed: 22.4 chunks/second
   - Bottleneck: CPU-bound (transformer model)
   - Opportunity: GPU acceleration can yield 5-10x speedup

3. FAISS INDEX BUILD (0.02 seconds for 50 PDFs)
   - Operation: Build IndexFlatL2 + add embeddings
   - Index type: Brute-force linear scan
   - Complexity: O(n) for add, O(n) for search
   - Negligible cost at 50 PDFs
   - Becomes bottleneck at 500+ PDFs (see V2 architecture)

4. SEARCH PERFORMANCE (3.32 ms P50)
   - Query encoding: ~2ms (single chunk embedding)
   - Index search: ~1ms (FAISS brute-force on 4,596 embeddings)
   - Total: ~3.32ms per query
   - Scales linearly with dataset size
   - At 500 PDFs: ~16-17ms per query (acceptable)


================================================================================
SCALING BOTTLENECKS (V1.5)
================================================================================

Issue 1: Build Time Grows Linearly
  - 50 PDFs:  236 seconds (~4 min)
  - 100 PDFs: 472 seconds (~8 min)
  - 500 PDFs: 2,362 seconds (~39 min)
  Impact: New PDFs can't be added incrementally
  V2 Solution: HDF5 + incremental FAISS updates

Issue 2: Search Latency Grows Linearly
  - 50 PDFs:  3.32 ms
  - 100 PDFs: 5.39 ms
  - 500 PDFs: 16.62 ms
  Impact: User experience degrades with dataset size
  V2 Solution: IndexIVFFlat reduces to ~50-100ms (logarithmic)

Issue 3: Memory Usage Grows Linearly
  - 50 PDFs:  ~50 MB (embeddings) + ~30 MB (chunks)
  - 100 PDFs: ~100 MB
  - 500 PDFs: ~500 MB
  Impact: Server memory constraints
  V2 Solution: HDF5 persistent storage + on-disk access

Issue 4: No Metadata Filtering
  - All searches scan entire dataset
  - Can't filter by paper, date, etc.
  Impact: Inefficient searches on large datasets
  V2 Solution: SQLite metadata + pre-filtering
